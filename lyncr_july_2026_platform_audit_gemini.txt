## Track A — Call Telemetry, Live Talk-Time & Recording Player

### A.1 Problem (pre-fix)

- Talk-time display logic was duplicated across lib/daily-call-telemetry.ts, routing-call-history-dialog.tsx, activity views, and routing-telemetry-cache.ts with inconsistent padding (e.g. 1:5 vs 1:05).
- HUD Daily Talk / Weekly Talk pills could appear to "stall" during active calls because:
  1. Baseline seconds only updated on call-completed Pusher event or REST refresh.
  2. scheduleRefreshBaseline() after hangup could overwrite optimistic increments with stale Neon reads (race).
  3. call-answered was not tracked in useRealTimeStats — no live tick while caller still on line.

### A.2 Solution — unified formatters

File: lib/telemetry-formatters.ts

Exports:
- formatSecondsToClock(sec) → m:ss or h:mm:ss; "0:00" for null/NaN
- parseTalkSecondsFromDisplay(display) → seconds from "12:05" or "1:02:03"

Wired into:
- lib/daily-call-telemetry.ts (formatTalkTime, formatTalkDuration, formatAvgTalkTime all delegate here)
- lib/routing-telemetry-cache.ts (re-exports parseTalkSecondsFromDisplay)

Tests: tests/telemetry-formatters.test.ts

### A.3 Solution — live talk-time counters

File: lib/hooks/use-real-time-stats.ts

Changes:
1. ActiveCallSession now includes answeredAt (set on Pusher call-answered).
2. Subscribes to call-answered on owner-{userId} channel (in addition to call-initiated / call-completed).
3. liveTalkTick interval (1s) while any session has answeredAt.
4. inProgressTalkSeconds = sum of (now - answeredAt) for active answered legs.
5. liveDailyTalkSeconds = dailyTalkSeconds + inProgressTalkSeconds
6. liveWeeklyTalkSeconds = weeklyTalkSeconds + inProgressTalkSeconds
7. refreshBaseline merge: Math.max(prev, apiValue) for talk seconds — prevents stale API from resetting HUD after optimistic call-completed increment.

File: components/dashboard/routing-telemetry-strip.tsx
- Displays liveDailyTalkSeconds / liveWeeklyTalkSeconds (not raw baseline).
- Passes live values to RoutingCallHistoryDialog expectedTalkSeconds for HUD parity checks.

Pusher events consumed (owner channel):
| Event | HUD effect |
|-------|------------|
| call-initiated | dailyCalls++, active session added |
| call-answered | session.answeredAt set → live tick starts |
| call-completed | session removed; talkSec added to baseline; debounced REST refresh |

### A.4 Solution — inline recording player

File: components/dashboard/CallAnsweredModal.tsx

ActiveCallRow extended (lib/hooks/use-active-call-form.ts):
- recording_url?: string | null

Behavior:
- callLogRowFromApi + showCallRow merge preserve recording_url from API rows.
- Poll GET /api/calls/answered-recent?withinMinutes=30 every 4s until recording_url lands (Telnyx recording-status webhook is async).
- When recording_url present, renders HTML5 audio row in sheet header:

  <audio src={recording_url} controls className="h-8 w-full accent-cyan-400" />

Note: recording URLs arrive after answer via POST /api/voice/telnyx/recording-status (or Call Control equivalent). Manual intake rows typically have no recording.

Gap: OwnerCallCompletedPayload and getCallLogSnapshotForTelemetry do not yet include recording_url — polling is the current bridge.

## Track B — Unified Lead Salvage Pool & Recovery SMS

### B.1 Problem (pre-fix)

- Price-shopper / hang-up leads lived in two silos:
  1. ai_leads with disposition PRICE_REJECTED + is_salvageable (receptionist portal)
  2. lost_leads table from owner intake sheet (POST /api/leads/lost)
- Recovery cron called sendTelnyxSms directly without 10DLC failure classification.
- No single owner dashboard queue; no realtime alert when carrier blocks recovery SMS.

### B.2 Solution — unified salvage pool

File: lib/salvage-pool.ts

SalvagePoolEntry normalized shape:
- source: ai_lead | lost_lead
- caller_e164, summary, failure_reason, status, last_quoted_price_cents
- manual_retry_required, recovery_blocked_reason (when status=failed_10dlc)
- vehicle_label, service_type, call_log_id, collected JSONB

listUnifiedSalvagePool(ownerUserId, limit):
- Parallel fetch: listSalvageableLeads (ai_leads) + listLostLeadsForSalvagePool (lost_leads)
- Merge sort by created_at DESC
- Returns counts: ai_lead, lost_lead, manual_retry

API routes:
- GET /api/leads/salvage-pool — standalone JSON feed
- lib/server/leads-workspace-data.ts — includes salvageLeads in workspace SSR payload

UI: components/workspace-views/leads-workspace-view.tsx
- LeadSalvageSection — amber cards, tel: links, badges:
  - "Price rejected" (ai_lead)
  - "Lost lead" (lost_lead intake)
  - "SMS blocked — call back" (failed_10dlc / manual_retry_required)
- Shows recovery_blocked_reason when 10DLC blocked

### B.3 Solution — 10DLC-safe recovery SMS

File: lib/lost-lead-recovery-sms.ts

sendLostLeadRecoverySms(row):
1. generateLostLeadRecoverySms(row) — OpenAI gpt-4o-mini optional; template fallback
2. sendTelnyxSms via lib/telnyx-sms.ts
3. classifyTelnyxSmsError on failure

Outcomes:
| Result | DB status | Pusher |
|--------|-----------|--------|
| SMS sent | recovery_sent | (none extra) |
| 10DLC block | failed_10dlc | salvage-recovery-blocked + lead-salvageable |
| Other error | lost_lead (retry eligible) | — |

Pusher event salvage-recovery-blocked (NEW):
- Payload: lost_lead_id, phone_number, error, manual_retry_required, attempted_sms_preview
- Owner should call back manually or use approved SMS channel

Cron: GET /api/cron/recover-leads
- Now uses sendLostLeadRecoverySms() (not inline Telnyx)
- Response includes blocked_10dlc count

Tests: tests/lost-lead-recovery-sms.test.ts

### B.4 Database dependency

Migration 84: scripts/084-lost-leads-recovery.sql (lost_leads table)
Migration 85: scripts/085-manual-call-log-bridge.sql (manual_intake call_logs)

OWNER ACTION: Run 084 + 085 in Neon SQL Editor per scripts/MIGRATE-ALL.md.

Cron gap: vercel.json does NOT yet include /api/cron/recover-leads — only /api/voice/warm and /api/sms/flush-scheduled. Owner must add Vercel Cron + CRON_SECRET.

### C.1 Problem (pre-fix)

- Prior modernization report gap #3: "Pricing matrix not connected to owner-configurable rate cards."

### C.2 Solution — rate card loader

File: lib/service-rate-card.ts

Storage: onboarding_profiles.service_rules JSONB text column

Expected JSON shape (embedded in service_rules):
{
  "rate_card": {
    "version": 1,
    "services": { "lockout": 8500, "key_gen": 17500, ... },
    "vehicle_age_tiers": [ { "min_age_years": 12, "cents": 1500 }, ... ],
    "premium_makes": ["BMW", "Mercedes-Benz", "Tesla", ...],

Plain-text service_rules (non-JSON) → falls back to DEFAULT_SERVICE_RATE_CARD.

- buildIntakePricingMetadata({ quote, vehicle, rateCardSource }) → IntakePricingMetadata

API: GET /api/service-quote/rate-card
- Returns { rate_card, source: "onboarding_profiles.service_rules" | "default" }

## Cross-Track Data Flows

### Flow 1 — Live call → ticking HUD → recording playback

Telnyx answer → Pusher call-answered → useRealTimeStats sets answeredAt
  → liveDailyTalkSeconds ticks every 1s in routing telemetry strip
Telnyx hangup → Pusher call-completed → baseline += duration; session removed
  → refreshBaseline with Math.max merge (no stall/regression)
Telnyx recording-status → call_logs.recording_url
  → CallAnsweredModal poll finds URL → inline <audio controls>

### Flow 2 — Price decline → salvage pool → recovery SMS

CallAnsweredModal "Customer declined" → POST /api/leads/lost
  → INSERT lost_leads → Pusher lead-salvageable
  → Leads workspace LeadSalvageSection (via listUnifiedSalvagePool)
  → [20+ min] GET /api/cron/recover-leads → sendLostLeadRecoverySms
  → success: status recovery_sent | 10DLC: status failed_10dlc + salvage-recovery-blocked

### Flow 3 — Mobile dispatch from hopper

Mobile: tap JobPoolCard → mobileAssignRequest set
  → MobileTechnicianAssignOverlay opens
  → optional: tap hour cell to set time slot
  → double-tap technician → POST /api/owner/jobs/pool/{id}/schedule
  → Pusher job-booked → scheduler refreshes

### Flow 4 — Owner rate card → quote → booked job audit

Intake sheet open → GET /api/service-quote/rate-card
  → calculateServiceQuote with owner card
  → Confirm & book → POST /api/jobs/create
  → createUnassignedJobFromIntake stores pricing_metadata in ai_leads.collected

## Known Gaps & Recommendations for Gemini Review

1. CRON NOT SCHEDULED — Add /api/cron/recover-leads to vercel.json crons (every 15–30 min) + CRON_SECRET in production.

2. NEON MIGRATIONS 084/085 — Must be run manually; lost lead logging and manual call_log bridge fail gracefully with console warnings but features are degraded without them.

3. RATE CARD EDITOR UI — JSON in onboarding_profiles.service_rules works but no self-serve Settings form; recommend admin or owner pricing screen.

4. RECORDING URL LATENCY — 4s poll in CallAnsweredModal; consider Pusher call-recording-ready event when recording-status webhook fires.

5. SALVAGE REALTIME — lead-salvageable and salvage-recovery-blocked published but leads-workspace-view primarily SSR/SWR; add client Pusher bind for instant salvage card appearance.

6. MOBILE ASSIGN HOUR — Pool card tap uses next whole hour default; user may want explicit hour picker in overlay before double-tap confirm.

7. DOUBLE-TAP DISCOVERABILITY — 450ms double-tap on technician rows may be non-obvious; consider explicit "Confirm assign" button as accessibility fallback.

8. TALK TIME BASELINE vs LIVE — History dialog summary totals from call_logs may lag live HUD by in-progress seconds until hangup; documented in HUD comparison footnote.

9. AI_LEAD vs LOST_LEAD dedup — Same phone could appear twice if receptionist logs PRICE_REJECTED and owner logs lost_leads for same call; consider dedupe key on caller_e164 + day.

10. PRIOR REALTIME AUDIT GAPS UNCHANGED — Platform Admin still has no Pusher; no RTP/MOS quality; receptionist WebRTC end-call sync; see lyncr_realtime_call_audit_gemini.txt sections 7–9.

11. OPENAI RECOVERY COPY — Optional; template always available when OPENAI_API_KEY unset.

12. 10DLC — All outbound recovery SMS requires approved workspace campaign (same as lead alerts).
